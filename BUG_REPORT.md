## CRITICAL Bugs

### BUG-001 · Memory consolidation deletes data without creating a replacement

**File:** `app/services/memory_cleanup.py:487–512`
**Function:** `_execute_consolidation`

The `execute_memory_cleanup` tool with `actions_to_execute: ["consolidation"]` **permanently deletes every memory in each consolidation group without first creating a merged replacement**. This is an acknowledged half-implementation (a `TODO` comment exists in-line):

```python
# For now, we'll just delete the individual memories
# In a full implementation, you'd create a new consolidated memory
for memory_id in memory_ids:
    success = await self.vector_store.delete_memory(memory_id)
```

The API response reports `"consolidated_groups": N`, which the user reads as success, while their data is gone.

**Impact:** Irreversible data loss. The README advertises "Consolidate related memories" as a feature.

**Fix:** Inject `EmbeddingService` into `MemoryCleanupService`, generate a new embedding for a synthesised text, store it, and only then delete the originals. Return the new memory ID.

### BUG-002 · Hardcoded 384-dimension dummy embedding breaks cleanup for any non-MiniLM model

**File:** `app/services/memory_cleanup.py:236`

```python
dummy_embedding = [0.0] * 384  # Assuming 384-dim embeddings from MiniLM
```

`_get_all_memories()` queries ChromaDB using a zero-vector of length 384 to retrieve all records. If the user has selected a 768-dimension model (e.g. `all-mpnet-base-v2`, `bge-base-en-v1.5`) the query will fail with a ChromaDB dimension-mismatch error, and the exception is silently swallowed:

```python
except Exception as e:
    logger.error(f"Error getting all memories: {e}")
    return []   # ← returns empty list; cleanup analysis sees 0 memories
```

The user receives an analysis report saying "no duplicates found" on a non-empty database.

**Fix:** Read the dimension from the active embedding model config (`ServerConfig.embedding_model_name`) rather than hard-coding it.

### BUG-003 · TUI search is silently disabled — returns an empty list with no notification

**File:** `app/tui/memory_manager.py:399–406`

```python
async def search_memories(self, query, limit=10, filter_tags=None):
    """Search disabled - returns empty list."""
    return []
```

The Memory Management screen exposes no in-TUI search today, but callers expecting results receive an empty list with no error or warning. If a future UI element wires this up, users will see blank results and assume there are no matching memories.

**Fix:** Either raise `NotImplementedError` so callers fail loudly, or implement the search by delegating to `VectorStore.search_similar()`.

### BUG-004 · Embedding dimension mismatch when switching models corrupts future cleanup

**Files:** `app/storage/vector_store.py:33–64`, `app/services/memory_cleanup.py:236`

ChromaDB stores all vectors in a single collection. If the user switches from `all-MiniLM-L6-v2` (384-dim) to `all-mpnet-base-v2` (768-dim), new memories are stored with 768-dim embeddings while old ones have 384-dim embeddings. ChromaDB will raise an error on any subsequent query. There is no migration path, no warning in the TUI, and no guard in the storage layer.

**Fix:** Detect dimension mismatch on startup; warn the user and offer to re-index the database.

## HIGH Severity Bugs

### BUG-005 · Memory edit loses the original ID and creation timestamp

**File:** `app/tui/memory_manager.py:323–350`
**Function:** `update_memory_sync`

"Edit" is implemented as delete-then-add:

```python
def update_memory_sync(self, memory_id, content, ...):
    delete_result = self.delete_memory_sync(memory_id)
    return self.add_memory_sync(content=content, ...)  # returns NEW id
```

Every edit produces a brand-new UUID and a new `created_at` timestamp. Any external reference to the old ID (in logs, in other tools, in the MCP history) breaks silently. The user sees the memory updated in the TUI but is unaware the ID changed.

**Fix:** Use `VectorStore.update_memory()` (which already exists but calls the same delete/add pattern). A proper fix is to preserve the UUID and update metadata in-place through ChromaDB's `Collection.update()` API. The `created_at` timestamp must be explicitly preserved.

### BUG-006 · Delete confirmation window is 2 seconds — too short, and resets on navigation

**File:** `app/tui/screens/memory_management.py:105, 130–134`

```python
self._delete_confirm_timeout: float = 2.0

def on_screen_resume(self) -> None:
    self._reset_pending_delete()   # fires when returning from any modal
```

The 2-second window to press `d` a second time starts the moment the first `d` is pressed. The confirmation message overwrites the memory content in the table row (so the user can no longer see what they are about to delete). If the user glances up or takes 2.1 seconds to re-read, the confirmation silently expires. Returning from any modal (e.g. opening Edit then immediately closing it) also resets the state.

**Fix:** Use a modal dialog for destructive confirmation instead of a timed double-keypress. Show the full memory content in the dialog.

### BUG-007 · Error messages for Importance field are ambiguous

**File:** `app/tui/screens/modals/add_memory.py:88–94`

```python
except ValueError:
    self.app.notify("Importance must be a number between 0 and 10", severity="error")
```

The same `except ValueError` catches both `float("abc")` (not a number) and the range check `not 0 <= importance <= 10`. The user cannot tell whether they entered non-numeric text or a valid number out of range.

**Fix:** Catch the `float()` conversion separately and show distinct messages, e.g. "Importance must be a number" vs. "Importance must be between 0.0 and 10.0".

---

### BUG-008 · Storage path change takes effect without validating writability

**Files:** `app/tui/settings/onboarding_screen.py:94–99`, `app/tui/screens/modals/storage_config.py:65–76`

The user can enter any path (e.g. `/root/no_permission`, `../../etc`) and the app persists it to `.env` via `ConfigManager.update_env_variable()`. The path is never checked for existence, writability, or whether it is absolute. The failure happens later, on the next app restart, with a confusing exception from ChromaDB or `pathlib`.

**Fix:** After expanding the user path, attempt `Path(expanded_path).mkdir(parents=True, exist_ok=True)` and a test write before persisting.

---

### BUG-009 · Corrupted ChromaDB has no recovery path

**Files:** `app/storage/vector_store.py` (constructor), `app/tui/memory_manager.py` (constructor)

If the ChromaDB directory (`~/.yaade/chroma/`) is corrupt (partial write, disk full during index flush, etc.), the `chromadb.PersistentClient()` call raises an exception. This exception propagates up through `memory_manager.py` and crashes the entire TUI with a raw Python traceback. There is no user-facing error message, no suggestion to re-initialise, and no documentation on what to do.

**Fix:** Wrap the `PersistentClient` constructor in a try/except; present a clear error screen with the option to reinitialise (move the corrupted directory aside and start fresh).

---

### BUG-010 · Embedding generation failure shows a cryptic internal error to the user

**File:** `app/tui/memory_manager.py:110–116`

```python
except Exception as e:
    return {"error": f"Failed to generate embedding: {str(e)}", "status": "failed"}
```

The returned dict propagates up to the screen which shows the raw exception string (e.g. `OSError: [Errno 28] No space left on device`) in a `notify()` toast that auto-dismisses. The user doesn't know the memory was not saved.

**Fix:** Translate common exception types (disk full, model not loaded, dimension mismatch) into human-readable messages; for model-not-loaded errors, guide the user to run `yaade download-model download <name>`.

## MEDIUM Severity Issues

### BUG-011 · Theme changes on highlight, not on confirm — Cancel doesn't revert

**File:** `app/tui/screens/memory_management.py:468–479`
(ThemeSelectScreen `on_theme_highlighted` handler)

```python
@on(OptionList.OptionHighlighted, "#theme-list")
def on_theme_highlighted(self, event):
    self.app.theme = self.selected_theme  # applied immediately on arrow key
```

Pressing the arrow keys to browse themes immediately applies the theme to the live UI. Pressing **Cancel** closes the modal but the theme is already changed — it is not reverted. This is an unexpected side-effect of navigation.

**Fix:** Apply the theme only on **Apply**/**Enter**. On **Cancel**, restore `self.app.theme` to the value it had when the screen opened.

### BUG-012 · ConfigManager silently swallows `.env` write failures

**File:** `app/tui/utils/config_manager.py:50–93`

```python
except Exception:
    return False
```

When `.env` write fails (disk full, read-only filesystem, permission denied), the function returns `False`. Most callers check the return value and show a success notification anyway, because the call-site condition is `if ConfigManager.update_env_variable(...)` only for some paths. Users believe their setting was saved when it was not.

**Fix:** Log the exception before returning `False`. Show a persistent error notification in all callers.

### BUG-013 · Memory list is hard-capped at 50 entries with no pagination

**File:** `app/tui/screens/memory_management.py:200`

```python
self.memories = await app.manager.list_all_memories(limit=50)
```

Users with more than 50 memories cannot see the rest in the TUI. There is no scroll indicator, no "showing 50 of N" count, and no way to load more. The statistics panel does show the total count, creating a discrepancy.

**Fix:** Add pagination (Load More button or infinite scroll via keyboard shortcut) or at minimum increase the limit and show "X of Y memories shown".

```python
display_content = content[:200] + "..." if len(content) > 200 else content
```

Memories longer than 200 characters are truncated in the table. The only way to see the full content is to open the Edit dialog, which is misleading (implies intent to modify). There is no read-only detail view.

### BUG-017 · No warning before storage path change makes existing memories inaccessible

**File:** `app/tui/settings/settings_screen.py:280–293`

Changing `YAADE_DATA_DIR` to a new (empty) directory does not migrate existing ChromaDB data. Old memories become inaccessible. The user is not warned that this is a destructive change.

**Fix:** Show a confirmation dialog explaining that existing memories will not be migrated, and optionally offer a migration path.

### BUG-018 · Setup script execution has no audit trail or error persistence

**File:** `app/tui/settings/setup_screen.py:277–294`

Setup script results are shown as a transient `notify()` toast. If setup fails, the user must re-run it to see the error again. There are no logs written and no persistent status indicator.

**Fix:** Write setup results to a log file under `~/.yaade/setup.log` and show a persistent status indicator in the Setup screen.

### BUG-019 · Tag input has no limits (count or length)

**Files:** `app/tui/screens/modals/add_memory.py:86`, `app/tui/screens/modals/edit_memory.py:108`

Tags are split on commas with no validation of count or per-tag length. A user (or an AI agent) could accidentally submit 100 long tags, filling ChromaDB metadata beyond its practical limits.

**Fix:** Enforce a maximum of 20 tags and a maximum tag length of 64 characters; show a counter in the input field.

### BUG-020 · Embedding model download has no progress feedback

**File:** `app/tui/screens/modals/embedding_model_select.py:452–495`

The download worker is fired as a background task. The user sees "Downloading…" but no percentage, ETA, or file size. Large models (420 MB+) can take several minutes on a slow connection.

**Fix:** Stream download progress from `huggingface_hub` and update a progress bar widget.

### BUG-021 · Network failures during model download cannot be retried from the UI

**File:** `app/tui/screens/modals/embedding_model_select.py:459–495`

If a download fails (network timeout, DNS failure), the error is shown in a notification that auto-dismisses. The user must re-open the modal and select the same model again to retry.

**Fix:** Show a persistent error message with a **Retry** button inside the modal.

### BUG-022 · No error recovery if the embedding model cache directory is missing

**File:** `app/tui/screens/modals/embedding_model_select.py:35–57`

If `SENTENCE_TRANSFORMERS_HOME` points to a directory that does not exist, the cache status check fails silently, incorrectly reporting all models as "not cached".

**Fix:** Create the directory if it does not exist, or show a clear "cache directory missing" warning.

### BUG-023 · `list_all_memories` returns no embeddings, making similarity search in cleanup unreliable

**File:** `app/services/memory_cleanup.py:226–255`

`_get_all_memories()` retrieves stored embeddings only if `results.get("embeddings")` is populated, but the ChromaDB query does not explicitly request `include=["embeddings"]`. Near-duplicate similarity is then computed from the stored metadata field (if any) rather than the actual embedding vectors.

**Fix:** Pass `include=["embeddings", "documents", "metadatas"]` to the ChromaDB query.


**File:** `app/tui/screens/modals/add_memory.py`, `edit_memory.py`

Users have no feedback about content length. Very short (1-character) or extremely long content is accepted silently.

### UX-003 · "Loading statistics…" text persists briefly after data loads

**File:** `app/tui/screens/memory_management.py:111`


The loading text is replaced only after `on_mount` completes, which can take 0.5–2 s on slow hardware. No spinner is shown.

### UX-004 · No keyboard shortcut to refresh the memory list

**File:** `app/tui/screens/memory_management.py` (BINDINGS)

`r` is documented in the README as the refresh shortcut, but it is not listed in the on-screen footer bindings.

### UX-005 · No undo for deletions

There is no soft-delete or trash mechanism. Once a memory is deleted (either manually or via cleanup), it is permanently gone.

**Recommendation:** Implement a 30-second undo window after deletion (hold the record in memory, then purge from ChromaDB only after the window expires).

### UX-006 · First-run experience unclear if no model is cached

When the app starts for the first time and no embedding model is cached, it downloads the default model automatically with no indication to the user. The TUI appears frozen until the download completes.

**Recommendation:** Show a first-run splash screen with a download progress bar before loading the main menu.

### UX-007 · Importance score default (1.0) is not shown in the Add Memory form

**File:** `app/tui/screens/modals/add_memory.py`

The importance field is blank by default. Users do not know what the default value is.

## Security Concerns

### SEC-001 · User-supplied storage path is not sanitised

**File:** `app/tui/screens/modals/storage_config.py:75–76`

`os.path.expanduser(new_path)` is called but the result is not canonicalised with `Path.resolve()`. An attacker (or a misconfigured AI agent) could supply `../../sensitive_dir`.

**Recommendation:** Call `Path(expanded_path).resolve()` and reject any path that does not start with the user's home directory or an explicit allowlist.

---

### SEC-002 · ConfigManager writes arbitrary values to `.env` without key validation

**File:** `app/tui/utils/config_manager.py:50–93`

The `key` parameter is written directly to the `.env` file. A newline-injected key could corrupt the file or inject environment variables.

**Recommendation:** Validate that `key` is in the set of known `YAADE_*` variables before writing.

---

### SEC-003 · Embedding model ID from `.env` is passed to HuggingFace download without validation

**File:** `app/tui/screens/modals/embedding_model_select.py:441–444`

If a user (or a compromised config file) sets `YAADE_EMBEDDING_MODEL_NAME` to an arbitrary value, it will be passed to `sentence_transformers.SentenceTransformer()`, which will download and execute arbitrary model code from HuggingFace.

**Recommendation:** Validate the model ID against the allowlist in `app/models/embedding_models.py`.
